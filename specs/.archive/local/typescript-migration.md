# TypeScript Migration

Migrate the entire wallfacer frontend from JavaScript to TypeScript, gradually,
without disrupting the existing no-bundler `<script>`-tag architecture or the
Go binary's `//go:embed ui` pipeline.

---

## Motivation

The frontend currently consists of ~92 hand-authored JS modules plus ~90
Vitest test files in `ui/js/`. All modules run in the browser global scope —
there are no ES modules or bundler. Over time this has led to:

- No compile-time guarantees about function signatures across files.
- Silent regressions when handlers change shape (events, API payloads, DOM
  contracts).
- Heavy reliance on test coverage and runtime checks to catch type errors
  that a type system would refuse to compile.
- Editor IntelliSense that works only to the extent the globals happen to
  be in scope.

TypeScript would catch these at build time with near-zero runtime cost, while
the Biome lint pass can stay exactly the same.

## Scope

**In scope**
- Introduce a TypeScript toolchain in `ui/` (tsconfig, esbuild, typecheck).
- Migrate every hand-authored file under `ui/js/` and `ui/js/lib/` and
  `ui/js/office/` from `.js` to `.ts`.
- Migrate tests under `ui/js/tests/` from `.js` to `.ts`.
- Add `ui/types/globals.d.ts` declaring cross-file globals incrementally as
  modules migrate.
- Wire TS compile + typecheck into `make build` and `make test`.
- Update Biome and Vitest to lint/run `.ts` alongside `.js`.

**Out of scope**
- Converting to ES modules (`import`/`export` in `<script type="module">`).
  That is a separate, equally large refactor (rewrite every file plus
  `scripts.html`, and probably adopt a bundler). Treat as a follow-up spec.
- Rewriting `ui/js/vendor/*` (third-party: highlight, marked, sortable, xterm).
- Rewriting `ui/js/generated/*` (emitted by `scripts/gen-api-contract.go`).
  These will get companion `.d.ts` files instead.
- Adopting a frontend framework (React/Vue/etc).
- Rewriting Go API types as shared TS types (a future spec could generate
  `.d.ts` from `internal/apicontract`, but this migration leaves
  `generated/types.js` as-is and adds `.d.ts` shims).

## Design

### Architecture: TS source in place, `.js` in a gitignored build directory

Source files live at the same paths as today but with a `.ts` extension.
Compiled output lives under `ui/js/build/`, mirroring source paths:

```
ui/js/lib/clipboard.ts              # source of truth (committed)
ui/js/build/lib/clipboard.js        # build artifact (gitignored)
```

**Only the `.ts` is committed.** The `ui/js/build/` tree is gitignored
and regenerated by `node scripts/build-ts.mjs` (invoked via `make ui-ts`).
This keeps `ui/js/` unambiguously hand-written source and prevents
source/output drift from committed artifacts. `<script>` tags in
`ui/partials/scripts.html` reference the build path directly (e.g.
`/js/build/lib/clipboard.js`), and the Go static handler at `/js/` serves
the nested path without change.

The `//go:embed ui` directive picks up `ui/js/build/` at Go build time, so
as long as `make ui-ts` runs before `go build`, the embedded binary
contains the compiled output. The Makefile's `build` target chains
`fmt → lint → ui-ts → build-binary`, enforcing this ordering locally.
CI workflows (`release-binary.yml`, `release-desktop.yml`, `test.yml`)
that bypass the Makefile must explicitly run `node scripts/build-ts.mjs`
between `npm ci` and the Go/Wails/vitest invocation — see each workflow
for the exact step.

The `//go:embed ui` directive still embeds `.ts` source files into the
binary alongside the compiled `.js` (the directive cannot easily filter
extensions). At current sizes this is ~200–300 KB of dead weight, which
is acceptable. Reassess after migration completes.

Rejected alternative: commit the compiled `.js` alongside the `.ts`
(the previous design in this spec). Rejected because committed generated
artifacts drift silently when someone edits the `.js` by hand, and
because the precedent it cited (`ui/js/generated/routes.js`) is really
"generated from Go" — a compile-once-per-schema-change artifact that
doesn't re-emit on every edit. TypeScript output re-emits on every
source edit, so committing it creates a churn asymmetry that isn't
worth the "fresh `go build` works without Node" property. `make build`
and CI guarantee Node runs before `go build`, which covers the
intended use cases.

Rejected alternatives:

| Option | Why rejected |
|---|---|
| `ui/src/*.ts` → compile to `ui/js/*.js` | Requires restructuring directories AND adding TS simultaneously. Two risks at once. |
| `frontend/src/*.ts` → compile to `ui/js/*.js` | Most disruptive. Touches every path reference across Go handlers and templates. |
| JSDoc + `checkJs: true` | Gives ~70% of the type safety with 0% churn, but stops short of real refactors (generics, interfaces, narrow types). Rejected in favor of full TS. |
| Big-bang rewrite | 92 modules + 90 tests at once. Blocks all other frontend work for the duration, produces an un-reviewable diff. |

### Build pipeline

Two tools, one each for speed and correctness:

- **esbuild** transpiles `.ts` → `.js` in place. Fast (tens of ms), zero
  config, preserves the per-file global-scope semantics via
  `--format=iife --global-name=...` when needed (most files don't need a
  wrapper because they declare top-level `function`/`var` that end up in
  the script's global scope under the no-module model).
- **tsc --noEmit** runs the full type checker. Slow but correct. Runs in
  `make lint` and in CI.

`esbuild` is installed as a `devDependency` in `ui/package.json` along with
`typescript`. The Makefile orchestrates both. There is no `npm run build`
step exposed to end users — the Makefile is the single entrypoint.

### Global-scope semantics

Because the runtime model is still "every script dumped into the window
global scope" (not ES modules), each compiled `.js` file must expose the
same top-level names it does today. esbuild's default behavior when given a
`.ts` file with top-level `function foo(){}` is to emit exactly that — no
IIFE, no `export`. Verified with the pilot.

`ui/types/globals.d.ts` declares the names that other files rely on
without `import`. Example:

```ts
// ui/types/globals.d.ts
declare function copyWithFeedback(
  text: string,
  btn: HTMLElement | null,
  feedback?: string,
  duration?: number,
): void;
```

This file grows monotonically as migration progresses. When every module
has migrated, `globals.d.ts` is a complete type index of the frontend's
public API.

### Test loading

Tests use `vm.runInContext` against raw `.js` source read from disk (see
`ui/js/tests/lib-deps.js`). After migration they read the **compiled** `.js`
output, so:

- `make test-frontend` depends on `make ui-ts` to ensure compiled output
  exists before Vitest runs.
- `ui/js/tests/` files themselves migrate to `.ts`. Vitest natively runs
  `.ts` test files via esbuild, no extra config required.
- `lib-deps.js` becomes `lib-deps.ts` but continues to reference the
  compiled `.js` files (that is what `vm.runInContext` needs).

### Biome and Vitest config changes

```json
// ui/biome.json — lint both .js and .ts with the same rule set
"files": {
  "ignore": [
    "js/vendor/**",
    "js/generated/**",
    "css/vendor/**",
    "node_modules/**"
  ]
}
```

Biome 1.9 lints `.ts` natively and shares configuration with `.js`, so no
additional setup is needed. The compiled `.js` twins of migrated `.ts`
files remain lintable — esbuild's transpiled output is
Biome-compatible (verified on the clipboard pilot).

```js
// ui/vitest.config.js — accept .ts tests
test: {
  include: ["js/tests/**/*.test.{js,ts}"],
  coverage: {
    include: ["js/**/*.{js,ts}"],
    exclude: ["js/vendor/**", "js/tests/**"],
  }
}
```

### Makefile changes

Add a `ui-ts` target that invokes `node scripts/build-ts.mjs`. Wire it
into `build` and `test-frontend` so compiled `.js` artifacts exist
before Go embeds them or Vitest loads them.

```make
# Compile TypeScript sources under ui/js/ into ui/js/build/.
ui-ts:
	cd ui && node scripts/build-ts.mjs

# Run strict type-check without emitting output.
typecheck-js:
	cd ui && npx --no-install tsc --noEmit || \
		(cd ui && npm install --no-audit --no-fund && npx --no-install tsc --noEmit)

build: fmt lint ui-ts build-binary pull-images
lint-js: typecheck-js  # followed by biome
test-frontend: ui-ts
	cd ui && npx vitest@2 run
```

### CI workflows

Three workflows run `go build`/`wails build`/`vitest` directly without
going through `make`, so they must explicitly compile TypeScript before
those invocations:

- `.github/workflows/release-binary.yml` — before each matrix cross-compile.
- `.github/workflows/release-desktop.yml` — before `wails build` on
  macOS/Windows/Linux runners.
- `.github/workflows/test.yml` — in `test-linux`, between `npm ci` and
  the Go/vitest steps. (`test-macos`/`test-windows` only build and
  vet Go; they don't exercise UI assets so they don't need the step.)

Each inserts `setup-node@v4` with `cache: npm` keyed on
`ui/package-lock.json`, `npm ci` in `ui/`, then
`node scripts/build-ts.mjs`. Cached installs keep the overhead to a
few seconds per run.

### Git tracking

Only `.ts` sources are committed. `ui/js/build/` is gitignored (entry
added to the repo `.gitignore`). This makes `ui/js/` unambiguously
hand-written source and removes the drift risk of committed generated
artifacts. `//go:embed ui` still embeds `ui/js/build/` because
`make ui-ts` (local) and the CI workflow steps (remote) populate it
before `go build` runs.

A CI freshness test like `internal/apicontract/generate_test.go` is
**not** applicable to this layout — there is no committed `.js` to
compare against. The equivalent guard is that CI workflows fail to
produce a working binary if `build-ts.mjs` is skipped, which is
prevented by the explicit workflow steps above.

## Migration order

1. ✅ **Pilot** — `ui/js/lib/clipboard.ts` (completed). Demonstrated the
   full toolchain end-to-end. Output landed as `ui/js/lib/clipboard.js`
   under the original "committed build artifact" design.
2. ✅ **Build directory refactor** — moved compiled output from sibling
   `.js` files to the gitignored `ui/js/build/` tree. Updated `scripts.html`,
   `lib-deps.js`, esbuild driver, lint/format ignore lists, ideate churn
   filters, and the three CI workflows that invoke `go build`/`wails build`/
   `vitest` outside of `make`.
3. **Rest of `ui/js/lib/`** — 10 remaining utility files (`config-toggle`,
   `floating-toc`, `formatting`, `log-stream`, `markdown-render`,
   `modal`, `modal-controller`, `scheduling`, `tab-switcher`, `toggle`).
   Few cross-deps; safe to batch in one PR.
4. **Leaf app modules** — `theme.js`, `events.js`, `state.js`, `markdown.js`,
   `oversight-shared.js`, etc. (no or few callers).
5. **Mid-tier modules** — `api.js`, `render.js`, `tasks.js`, `modal-*.js`.
   Each conversion grows `globals.d.ts`.
6. **Entry modules** — `command-palette.js`, `planning-chat.js`,
   `spec-mode.js`, `terminal.js`, `explorer.js`.
7. **`ui/js/office/`** — pixel-art module. Self-contained; can migrate any
   time in parallel with step 5+.
8. **Tests** — convert `ui/js/tests/**/*.test.js` → `.test.ts`. Vitest
   handles `.ts` natively. Last, so test infrastructure stabilizes.
9. **Cleanup** — generate `.d.ts` companions for `generated/types.js`
   and `generated/routes.js`. Wire `make typecheck-js` (or `make lint-js`)
   into `test-linux` in `test.yml` so type regressions fail CI.

Every step is a standalone commit (or cohesive PR) that keeps `make test`
green.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| esbuild emits different top-level semantics than hand-written JS | Pilot verifies byte-for-byte equivalent top-level bindings. If esbuild ever wraps output in an IIFE, switch to `--format=esm` and add a post-processing pass, or use `tsc` emit with `target=es2022, module=none`. |
| `make build` breaks for devs who only run `go build` | `go build` succeeds but embeds an empty or stale `ui/js/build/`, causing the browser to 404 on `/js/build/*.js`. Document in CLAUDE.md and `docs/guide/configuration.md` that `make build` is now required. CI workflows insert an explicit `node scripts/build-ts.mjs` step. |
| Tests or Go handlers read compiled JS from a gitignored directory | `//go:embed ui` transparently picks up `ui/js/build/` when populated; empty dirs embed cleanly too. `make ui-ts` is a declared prerequisite of `make test-frontend` and `make build`. CI workflows replicate the step. |
| Embedded binary size grows | Measured ~200–300 KB at 46 source files. Acceptable. Reassess after migration. |
| `globals.d.ts` becomes a maintenance burden | Short term, yes. Long term, it is the catalog of the frontend's global surface area — valuable documentation. Freezes when migration completes. |
| Tests that `readFileSync` a specific JS file break | `ui/js/tests/lib-deps.js` references compiled outputs at `build/lib/*.js`. Update the path when migrating a file; `make test-frontend` chains `ui-ts` so the file always exists. |
| Vitest config `include` pattern mismatches | Tested in the pilot. |

## Acceptance criteria

- `ui/tsconfig.json` exists with `strict: true`, `noEmit: true`, targets
  browser DOM + ES2022, and excludes `js/vendor`, `js/generated`, `js/build`.
- `ui/types/globals.d.ts` exists and declares every cross-file global used
  by migrated files.
- `make ui-ts` compiles every `.ts` under `ui/js/` to a corresponding
  `.js` under `ui/js/build/`, mirroring source directory structure.
- `make typecheck-js` passes with zero errors.
- `make build` produces a working binary that serves the compiled JS.
- `make test` passes — all backend + frontend tests green.
- Every hand-authored JS file under `ui/js/` (excluding `vendor/`,
  `generated/`, and `build/`) has been converted to `.ts`.
- `ui/js/tests/` uses `.ts` throughout.
- Biome lints `.ts` with the same rule set as `.js` and ignores
  `js/build/**`.
- `.gitignore` includes `ui/js/build/`.
- Release workflows (`release-binary.yml`, `release-desktop.yml`) and
  `test.yml`'s `test-linux` job run `node scripts/build-ts.mjs` before
  any `go build`/`wails build`/`vitest` invocation.
- `test-linux` in `test.yml` invokes `make typecheck-js` (or equivalent)
  so TypeScript type errors fail CI.
- CLAUDE.md and `docs/guide/configuration.md` note the new build
  requirement.

## Not in this spec (follow-ups)

- **ES modules migration**: convert `<script>` tags to
  `<script type="module">`, replace globals with `import`/`export`, adopt a
  bundler (esbuild bundle mode or Vite). Larger scope, different risk
  profile.
- **Generated API types**: emit `internal/apicontract` as `.d.ts` so
  handlers and frontend share types. Worth a dedicated spec.
- **Strictness ratchet**: after initial migration, tighten `tsconfig` (e.g.
  `noImplicitAny`, `strictNullChecks` already on; consider
  `exactOptionalPropertyTypes`, `noUncheckedIndexedAccess`).

## Pilot (completed)

The pilot converted `ui/js/lib/clipboard.js` — a single 25-line function
with no cross-file references — and landed the full toolchain:

- `ui/tsconfig.json`
- `ui/types/globals.d.ts` (initially tiny)
- `ui/scripts/build-ts.mjs` (esbuild driver)
- `ui/package.json` devDeps (`typescript`, `esbuild`)
- Makefile `ui-ts`, `typecheck-js` targets
- Biome, Vitest, `.gitignore` updates
- `ui/js/lib/clipboard.ts` replacing `clipboard.js`

A follow-up refactor moved compiled output from `ui/js/lib/clipboard.js`
to `ui/js/build/lib/clipboard.js` and wired `node scripts/build-ts.mjs`
into the three CI workflows that bypass `make`.

After these land, subsequent migrations are mechanical: rename the file
to `.ts`, add type annotations, extend `ui/types/globals.d.ts`, update
`ui/partials/scripts.html` to point at `/js/build/<path>`, update
`ui/js/tests/lib-deps.js` if other tests depend on it, run
`make ui-ts test-frontend`, ship.
